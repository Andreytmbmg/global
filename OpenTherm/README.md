# 🌡️ Регулировка температуры комфорта OpenTherm — Глобальный сценарий

Глобальный сценарий SprutHub для автоматической регулировки температуры теплоносителя котла (OpenTherm) на основе комфортной температуры в нескольких комнатах, погодных условий на улице и текущего состояния окон/вентиляции/тёплого пола.

Скрипт не просто поддерживает фиксированную температуру подачи, а **динамически подбирает её**, ориентируясь на самую "проблемную" (обычно самую холодную) комнату в доме, с учётом прогноза погоды, ветра, солнечной инсоляции и стабильности отклонений во времени.

---

## 🎯 Основная идея

1. Рассчитывается **базовая температура теплоносителя** исходя из уличной температуры, прогноза на 6 часов вперёд, ветра, УФ-индекса и облачности (упрощённый аналог погодозависимой кривой отопления).
2. Для каждой комнаты сравнивается текущая температура с целевой (заданной на термостате).
3. Комнаты с открытым окном или включённой вентиляцией **исключаются** из расчёта, чтобы не разгонять котёл впустую.
4. Среди оставшихся комнат находится **самая холодная относительно своей цели** — по ней и корректируется температура теплоносителя.
5. Если отклонение в этой комнате сохраняется несколько циклов подряд (стабильно холодно/стабильно жарко), коррекция усиливается ступенчато.
6. Итоговая температура теплоносителя ограничивается диапазоном **40–80°C** и отправляется в характеристику котла.
7. Подробный отчёт по каждой комнате (температура, статус радиаторов/тёплого пола/окон/вентиляции, качество воздуха) отправляется в Telegram.

## 🧮 Логика расчёта базовой температуры теплоносителя

Отправная точка — упрощённая погодозависимая кривая:

```
baseWaterTemp = 53 + (0 − outsideTemp)
```

То есть при 0°C на улице базовая температура подачи — 53°C, и растёт при похолодании.

Далее применяются корректировки:

- **Прогноз на 6 часов** — если ожидается похолодание/потепление, температура подачи заранее подстраивается (`(forecastTemp − outsideTemp) / 2`).
- **Ветер** — усиливает теплопотери дома, добавляет +1…+3°C в зависимости от силы ветра (от 3 м/с и выше).
- **УФ-индекс и облачность** — солнечный нагрев снижает потребность в тепле, корректировка от −1 до −5°C в зависимости от интенсивности солнца и облачности.

Результат ограничивается диапазоном **40–80°C**.

## ❄️ Логика поиска "самой холодной" комнаты

- Комнаты с открытым окном или включённой вентиляцией помечаются как игнорируемые (`ignoreTemperature`) и не участвуют в расчёте.
- Из оставшихся комнат выбирается та, где разница `target − sensor` максимальна (самое большое отставание от цели).
- Если все комнаты либо игнорируются, либо уже теплее цели — берётся комната с минимальным превышением, чтобы избежать перетопа.

---

## 📈 Многоступенчатая коррекция по стабильности

Чтобы избежать резких скачков и "дребезга" при разовых колебаниях датчика, скрипт отслеживает, сколько циклов подряд самая холодная комната остаётся такой же:

- **Стабильно холодно** (Δ > 0.2°C несколько циклов подряд):
  - 3 цикла → +1°C к Δ
  - 4 цикла → +1.5°C к Δ
  - 5+ циклов → +2°C к Δ
- **Стабильно жарко** (Δ ≤ −0.2°C несколько циклов подряд):
  - 3 цикла → −1°C к Δ
  - 4 цикла → −1.5°C к Δ
  - 5+ циклов → −2°C к Δ

Дополнительно есть защитные проверки: если "стабильно жарко", но расчёт почему-то повышает температуру теплоносителя — коррекция принудительно понижается, и наоборот.

Если отклонение в самой холодной комнате близко к нулю (±0.2°C), температура теплоносителя сбрасывается на базовое значение.

## 🛡️ Отказоустойчивость

- Все обращения к устройствам идут через `safeGetValue()` — при недоступности аксессуара/сервиса/характеристики возвращается безопасное значение по умолчанию, а ошибка логируется, не прерывая выполнение сценария.
- При недоступности датчика температуры используется последнее известное значение (`global.lastRoomTemps`), а не значение по умолчанию, чтобы не "занижать" расчёт.
- При отсутствии данных о погоде (`global.yr`) используются последние сохранённые внешние данные (`global.lastExternalData`).
- Ошибки установки итоговой температуры теплоносителя и общие сбои в `adjust_heating()` перехватываются и дублируются уведомлением в Telegram.

## ⚙️ Требования к запуску

- Любое устройство для работы с целевой температурой газового котла отопления подключенное к **SprutHub** по MQTT, Zigbee и т.д.
- Платформа **SprutHub**, JS-движок Nashorn (ES5).
- Глобальная переменная `global.yr` с данными погоды (температура, прогноз на 6 часов, ветер, УФ, облачность) — предполагается, что заполняется отдельным сценарием интеграции с погодным сервисом.
- Настроенный канал уведомлений `Telegram_1`.
- Тип сценария — `GLOBAL`, точка входа — функция `global.adjust_heating()`, вызываемая по расписанию/триггеру из другого сценария (например, через `runGlobal(...)`).
- Для каждой комнаты необходимо создать вирутальный термостат откуда будут браться целевые значения температуры по комнатам.
